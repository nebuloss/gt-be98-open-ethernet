# 04 — Runner microcode LOAD ABI + datapath BRING-UP ORDER (BCM6813 XRDP)

## Purpose

This note reverses two things the open driver must get exactly right before a single
frame can move on real silicon:

1. **The Runner microcode LOAD ABI** — how the 8 per-core instruction images
   (`fw_binary_0..7`, 32 KB each) and prediction images (`fw_predict_0..7`, 1 KB each)
   are block-written into the per-core Runner instruction SRAM (`RNR_INST`) and
   prediction RAM (`RNR_PRED`): the per-core window addressing, the word count, the
   byte-swap, and the prediction-RAM packing.
2. **The datapath BRING-UP ORDER** — the numbered sequence of sub-block init the stock
   `data_path_init` performs (UBUS decode → CNPL → RNR core-addr → MPM/FPM → **ucode
   load** → scheduler → SBPM → BBH_RX → DMA → NAT-C → BBH_TX/XLIF → DSPTCHR → **QM
   enable 0x307** → UBUS-master enable → TCAM), plus the separate `drv_qm_init` that
   performs the QM SRAM `MEM_AUTO_INIT` + poll.

Gaps closed / de-risked:
- **M2** (open datapath bring-up) — pins the load loop and the ordering constraints.
- Audit `docs/audit/09-hardware-acceleration.md` §6 and `docs/audit/10-reimplementation-guide.md`
  M1a/M1b: the Runner core/image model and the RNR_INST window.
- Confirms/settles audit `08-hw-abi-regmap.md` §8 items **#11** (QM MEM_AUTO_INIT
  `0x138`/STS `0x13c`, done=bit0) and **#14** (per-subsystem core wakeup, not a
  `first_task` table), and CORRECTS one ordering detail in `runner_qm_init`.

All facts are from disassembly of the device's own `rdpa.ko`. Raw disasm is kept on the
build host under `~/re-scratch/rnr-load/` — this doc carries only offsets/encodings.

## ABI reference

### 1. Runner core / image model `[KO]`

- **8 packet-processor cores**, indexed 0..7. `ag_drv_rnr_inst_mem_entry_set` masks the
  core arg `& 0xff` and validates `core <= 7` (`0x1213a8: cmp w0,#7`).
- **Instruction SRAM = 8192 × 32-bit words per core** (32 KB). The entry accessor
  validates `index <= 0x1fff` (8191) (`0x1213a4: mov w3,#0x1fff`).
- **Per-core window base pointers** live in three relocated pointer arrays whose
  `+0x8` field is the base-pointer table (one mapped base per core, `[table + core*8]`):
  - `RNR_INST_BLOCK+0x8` → INST window bases
  - `RNR_PRED_BLOCK+0x8` → PRED window bases
  - `RNR_MEM_BLOCK+0x8`, `RNR_CNTXT_BLOCK+0x8` → data/context (used by cores_addr_init)
- A **runtime global at `.data+0x5d9b8`** holds the number of active cores; it gates all
  three load loops (`cmp; b.le` guards). = 8 on 6813.
- Two `.data` arrays parametrise the load:
  - `runner_core_size[core]` — instruction word count (8192 for a full image; may be 0
    to skip a core).
  - `fw_inst_binaries[core]` / `fw_pred_binaries[core]` — per-core source image pointers
    (8×8-byte pointer tables).

### 2. INST load loop — `drv_rnr_load_microcode` / `drv_rnr_load_instructions` `[KO]`

The load writes **directly to the ioremapped per-core INST window** (`*(RNR_INST_BLOCK+0x8)[core]`),
NOT through the `ag_drv_rnr_inst_mem_entry_set` register accessor (that accessor exists
but has **no caller in the load path** — it is autogenerated CLI/debug surface).

Per core (`runner_core_size[core]` words, source = `fw_inst_binaries[core]`):

| step | operation |
|---|---|
| unit | **two 32-bit words at a time** (`ldr x` = 8 bytes from source) |
| endian | `rev32 x` — byte-swap **each** 32-bit word independently (source images are stored byte-reversed vs the MMIO window) |
| store | `str x, [inst_base, (word_index>>1)*8]` → dest byte offset `= word_index*4` |
| count | loop `word_index` from 0 to `runner_core_size[core]` step 2 |

Net: for each source 32-bit word `w` at word index `i`, write `byteswap32(w)` to
`INST_base + i*4`. **Stride = 4 B/word, entry = 32 bits.** Equivalent to
`iowrite32be(native_le_word)` per word.

`drv_rnr_load_instructions(core, start_word, num_words, src)` is the same loop with a
caller-supplied `start_word` (must be even) and count — used for partial reloads.

### 3. PRED load loop — `drv_rnr_load_prediction` `[KO]`

Per core, source = `fw_pred_binaries[core]` (16-bit halfwords):

| step | operation |
|---|---|
| entry count | `runner_core_size[core] / 16` (= 512 for an 8192-word core → 512 pred entries) |
| read | `ldrh w0,[src],#2` — one 16-bit source halfword |
| endian | `rev32 w0` (byte-reverse into the 32-bit slot) |
| store | `str w0,[pred_base],#4` — write a **32-bit** slot, **stride 4** |

So the prediction RAM is written as **512 × 32-bit slots**, each carrying one byte-reversed
16-bit prediction value. Source image = 512 × 2 B = 1 KB (matches `fw_predict_* = 1024 B`);
dest window = 512 × 4 B = 2 KB. Equivalent to `iowrite32be(native_le_u16)` per entry.

### 4. Per-core INST accessor addressing `[KO]`

`ag_drv_rnr_inst_mem_entry_set(core, index, value)`:
`base = (*(RNR_INST_BLOCK+0x8))[core]`; `stride = RNR_INST_MEM_ENTRY_REG+0x14`;
`*(u32*)(base + index*stride) = value`. The `+0x14` field of the register descriptor is
the **entry stride** (confirms the descriptor convention: `+0x08` offset, `+0x10` count-1,
`+0x14` stride). Bounds: `core<=7`, `index<=8191`.

**Per-core enable/wakeup** is NOT in this list as a `first_task` table write — it happens

done bits before ucode load.

### 6. QM SRAM MEM_AUTO_INIT — `drv_qm_init` (`0xad060`) `[KO]`

`MEM_AUTO_INIT` is **not** performed by `data_path_init`. It is the **last** step of the
separate `drv_qm_init`, whose call order is:

`update_fpm_budget` → `drop_counters_ctrl_{get,set}` → **`qm_fpm_base_addr_set`** →
`global_cfg_fpm_coherent_base_addr_set` → `fpm_ctrl_{get,set}` → `ddr_sop_offset_set` →
`fpm_mpm_enhancement_pool_{size_tokens,0_1_size_byte,2_3_size_byte}_set` →
`wred_profile_cfg_set` ×2 → `fpm_buffer_reservation_data_set` → `bb_route_ovr_set` →
`q_context_set` → **`mem_auto_init_set`** → **`mem_auto_init_sts_get`** (poll).

`drv_qm_init` is invoked from `qm_init` (`0x91620`, a per-block init entry, dispatched by
the block-init table — not an inline `data_path_init` call). It therefore runs during
module bring-up *before* `data_path_init` writes `ENABLE_CTRL=0x307`.

Register pin-down (from the accessors):

| register | offset | encode |
|---|---|---|
| `QM_GLOBAL_CFG_MEM_AUTO_INIT` | **`0x138`** (`str w1,[x2,#312]`) | write fields: bit0=EN(byte0), byte1<<8, byte2<<16 |
| `QM_GLOBAL_CFG_MEM_AUTO_INIT_STS` | **`0x13c`** (`ldr w2,[x2,#316]`) | **done = bit0** (`and w2,w2,#0x1`) |

Confirms audit §8 #11 exactly.

### 7. QM ENABLE_CTRL bitfield — `ag_drv_qm_enable_ctrl_set` (`0x111670`) `[KO]`

Register at **QM block offset `0x000`** (`str w1,[qm_base]`, no add). The accessor packs 5
one-bit fields from 5 byte args into these bit positions:

| field (arg byte) | bit |
|---|---|
| a (byte0) | **0** |
| b (byte1) | **1** |
| c (byte2) | **2** |
| d (byte3) | **8** |
| e (byte4) | **9** |

`0x307` = bits {0,1,2,8,9} all set ⇒ all five sub-enables = 1. This decodes the magic
`0x307` and **confirms** `QM_ENABLE_CTRL_STOCK = 0x307` at offset `0x000`.

## Unresolved

- **Numeric contents of `runner_core_size[]` and the num-cores global** were read as a
  model (word count / core count), not dumped as literals; taken as 8192 words × 8 cores
  from the image geometry (`fw_binary_*`=32 KB, `fw_predict_*`=1 KB). A `.data` byte dump
  would pin the exact per-core sizes (some cores could carry a shorter image).
- **`data_path_init` does not itself program the QM auto-init** — it relies on `drv_qm_init`
  (dispatched via the block-init table). The exact dispatch order between `qm_init` and
  `data_path_init` within module load was inferred from the two being distinct entry points,
  not traced through the block-init table walker.
- **Per-core scheduler cfg values** (`cfg_sch_cfg_set` ×8) and the RNR `gen_cfg`
  ZERO_DATA/CONTEXT immediates were not decoded here (this note targets the load + order).
- **BBH_TX QMQ / Route-A queue→core binding** is a separate (Route A) path; only the
  common `bbh_tx_...general_cfg_set` + `txthresh_set` appear in `data_path_init`. QMQ
  register values remain per `re-notes/realhw/11-route-a-egress-spec.md` (audit §8 #12).
- **Prediction-RAM exact hardware slot width** (whether the upper/lower half of the 32-bit
  slot is the live 16-bit value) is inferred from the `rev32 w`/`str w` pair; the driver's
  `iowrite32be(le16)` is byte-identical to the stock store, so functionally settled, but the
  HW field position within the slot was not independently confirmed.
